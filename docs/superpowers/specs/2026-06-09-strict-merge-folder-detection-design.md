# 严格合并目录识别修复设计

## 背景

当前程序会把 `06-03-HYX-NP图片项链-6单-13个` 误判为已经生成的合并目录。
原因是已生成目录正则允许使用连字符表达序号范围，例如 `1-3-0603-...`，
因此名称开头的日期 `06-03` 被错误解释成序号范围，项目在扫描阶段直接跳过。

日期识别还有一个关联问题：识别前会删除名称开头的 `数字-`，导致
`06-03-...` 被截断成 `03-...`，无法识别为日期 `0603`。

## 目标

- 已生成合并目录只允许使用 `~` 或 `～` 表达序号范围。
- `06-03-...` 必须作为普通来源文件夹处理，并识别日期为 `0603`。
- 保持现有品类、数量、排序、合并、执行、撤销、压缩和 GUI 行为不变。

## 识别边界

| 名称 | 预期行为 |
| --- | --- |
| `06-03-HYX-NP图片项链-6单-13个` | 普通来源，日期为 `0603` |
| `06-03-CSY-银翅膀图片项链-26单-33个` | 普通来源，日期为 `0603` |
| `1~3-0603-NP图片项链-10单-20个` | 已生成合并目录，跳过 |
| `1～3-0603-NP图片项链-10单-20个` | 已生成合并目录，跳过 |
| `1-3-0603-NP图片项链-10单-20个` | 普通来源，不再按合并目录跳过 |
| `10-0603-产品名-1单-1个` | 普通来源，日期为 `0603` |

## 修改设计

### 已生成合并目录判断

收紧 `GENERATED_MERGE_FOLDER_PATTERNS`：

- 保留 `1~3-...`。
- 保留全角波浪号形式 `1～3-...`。
- 保留项目当前支持的 `1+2+3-...` 列表形式。
- 删除 `1-3-...` 连字符范围形式。

配置中的 `already_processed.patterns` 当前只使用 `~`，无需扩大或增加兼容规则。

### 日期识别

`detect_dates()` 直接在完整外层名称中运行日期正则，不再预先删除开头的
`数字-` 前缀。

这样：

- `06-03-...` 可识别为 `0603`。
- `10-0603-...` 会识别其中的 `0603`。
- 真正的 `1~3-0603-...` 已经在扫描阶段作为生成目录跳过，不进入业务识别。

## 错误处理与安全边界

- 不修改 dry-run 默认行为。
- 不修改 apply 和 undo 的大写 `YES` 确认规则。
- 不修改目标目录和同名 zip 的冲突跳过规则。
- 不删除或覆盖任何来源文件夹、压缩包或目标文件。
- 修复只影响扫描阶段的目录跳过判断和日期提取。

## 测试设计

增加核心回归测试：

3. `1~3-0603-...` 和 `1～3-0603-...` 仍属于已生成合并目录。
4. `1-3-0603-...` 不再属于已生成合并目录。
5. `10-0603-...` 日期识别结果为 `0603`。
