# 执行历史详情设计

## 1. 目标

为 Windows 文件整理助手增加只读的“执行历史”页面，帮助用户查看每次实际执行整理的结果。

第一版只显示 `apply` 运行，不显示 `dry-run` 或 `undo-last`。历史页面以单次运行及其最终结果文件夹为核心，不提供跨运行搜索、统计图表、历史删除、路径打开或任意历史撤销。

历史记录最多保留最近 100 次 `apply`。写入第 101 次记录时，自动删除最旧记录及其撤销操作，该记录永久失去撤销资格。`rename_log.csv` 不参与裁剪，继续作为长期人工审计记录。

## 2. 用户体验

### 2.1 页面入口

在 GUI 左侧导航新增“执行历史”。

每次进入该页面时重新读取 `organizer_run_log.json`。页面停留期间不轮询文件，也不提供单独的刷新按钮。

进入页面后不默认选择任何记录，右侧显示：

```text
请选择一条执行记录
```

### 2.2 页面布局

页面采用左右布局：

- 左侧显示 `apply` 运行列表。
- 右侧显示当前所选运行的详情。
- 左侧保留可见，不因选择记录而切换到独立页面或弹出窗口。

左侧每条记录只显示：

- 执行时间。
- 根目录。
- 整体状态。

状态显示规则：

| 日志状态 | GUI 文案 |
| --- | --- |
| `success` | 成功 |
| `partial` | 部分成功 |
| `failed` | 失败 |
| `running` | 执行中断 |

`running` 仅表示程序未完成最终状态写入。GUI 不修改原日志状态。

### 2.3 最终结果列表

右侧以最终结果文件夹为主要查看单位，每行显示：

- 最终文件夹名称。
- 目标路径。
- 来源数量。
- 是否合并。
- 执行状态。
- 识别日期。
- 识别品类。
- 单量。
- 数量。
- 命中关键词。

结果行可展开。展开后显示：

- 该最终结果包含的所有原始来源项。
- 失败或跳过原因，仅在存在原因时显示。

失败和跳过原因不占用默认表格列宽。

### 2.4 空状态与异常状态

当日志不存在、日志为空或没有 `apply` 记录时，显示：

```text
暂无执行历史，完成一次执行整理后会显示在这里
```

程序不因打开历史页面而创建空日志。

当日志损坏或无法解析时：

- 显示可理解的错误原因。
- 显示日志绝对路径。
- 不展示任何部分解析结果。
- 不自动修复、备份、覆盖或重建日志。
- 不影响 GUI 的其他页面继续使用。

## 3. 运行日志结构

### 3.1 现有记录

`organizer_run_log.json` 继续作为运行记录和撤销数据的唯一来源。现有 `operations` 字段继续只保存撤销所需的底层操作。

每次新版 `apply` 在同一条 run 记录中增加 `history_snapshot`：

```json
{
  "run_id": "...",
  "mode": "apply",
  "root": "...",
  "time": "...",
  "status": "success",
  "operations": [],
  "history_snapshot": {
    "schema_version": 1,
    "results": [
      {
        "result_id": "...",
        "final_name": "...",
        "target_path": "...",
        "source_items": [
          {
            "original_name": "...",
            "source_type": "folder",
            "source_path": "..."
          }
        ],
        "merged": true,
        "date": "0501-0505",
        "category": "...",
        "orders": 10,
        "quantity": 20,
        "matched_keywords": [],
        "status": "success",
        "error_reason": ""
      }
    ]
  }
}
```

字段约束：

- `schema_version` 用于以后兼容快照结构变化。
- `result_id` 是一次 run 内稳定且唯一的结果标识，用于执行期间更新对应快照。
- `source_items` 保存 `original_name`、`source_type` 和执行前的 `source_path`，不从执行后的磁盘状态反推。
- `matched_keywords` 保存来源项命中关键词去重后的有序列表。
- `status` 使用 `pending`、`success`、`skipped` 或 `failed`。
- `error_reason` 只在跳过或失败时填写。
- 新版 run 明确写入 `mode: "apply"`；旧记录没有 `mode` 时按现有 apply 记录兼容。

### 3.2 快照生成

apply 确认后，在开始真实文件操作前，根据已经打印并确认的整理计划建立快照：

1. 每个 `PlanGroup` 对应一个历史结果。
2. 初始结果状态为 `pending`。
3. 来源项在快照建立时固定，不因后续移动或重命名改变。
4. 每个结果执行完成后更新为 `success`、`skipped` 或 `failed`。
5. apply 整体结束时保存最终 run 状态。

快照识别信息按最终分组计算：

- 日期使用现有分组日期规则，多个日期显示日期范围。
- 品类使用最终分组品类。
- 单量和数量合计所有来源项。
- 命中关键词按来源顺序去重。
- 来源数量为分组中的来源项数量。
- 合并状态只有在来源项超过一个且该分组按合并流程处理时为 `true`。

GUI 不重新计算这些值，也不读取当前磁盘内容修正历史结论。

### 3.3 结果状态

结果状态含义：

- `pending`：快照已创建，但程序终止前没有写入最终结果；GUI 显示“执行中断”。
- `success`：该最终结果所需的整理操作成功完成。
- `skipped`：因目标冲突或其他既有跳过规则未执行。
- `failed`：执行期间发生异常。

run 的整体状态继续沿用现有 apply 计算结果。GUI 不根据结果列表重新推导整体状态。

### 3.4 旧记录兼容

没有 `history_snapshot` 的旧版 `apply` 记录：

- 继续显示在左侧历史列表。
- 计入最近 100 条上限。
- 右侧不结合 `rename_log.csv`、整理报告或当前目录猜测详情。
- 选中后显示“旧版记录，详情不完整”。

旧记录保留现有 `operations` 时，仍按现有规则参与最近一次撤销查找，直到被 100 条裁剪机制淘汰。

## 4. 保留与裁剪

### 4.1 保留规则

`organizer_run_log.json` 最多保留最近 100 条 `apply` 记录。

裁剪规则：

1. 创建 apply run 和后续保存 run 时调用同一个裁剪函数。
2. 所有新旧 `apply` 记录统一计数。
3. 按日志中的运行顺序保留最新 100 条，时间字段只用于展示，不作为唯一排序依据。
4. 超出上限时从最旧记录开始删除。
5. 删除的是完整 run，包括 `history_snapshot`、`operations` 和撤销状态。
6. 被删除记录不再具备撤销资格。

当前日志只保存 apply run；如果以后引入其他 run 类型，裁剪逻辑只能限制 apply 数量，不得误删其他类型。

GUI 不显示保留上限提示。README 必须说明：

- 最多保留最近 100 次执行整理记录。
- 超出后自动删除最旧记录。
- 被删除记录不能再撤销。
- `rename_log.csv` 不受该限制。

### 4.2 写入安全

继续使用临时文件加原子替换的安全写入方式：

- 成功写入完整临时 JSON 后才替换正式日志。
- 写入失败时保留替换前的完整日志。
- 不留下半截 JSON。
- 不通过静默创建新日志来掩盖已有日志损坏。

历史快照和撤销操作属于同一条 run，必须通过同一份内存数据和同一次安全保存保持一致。

## 5. 模块边界

### 5.1 `file_helper.py`

负责：

- 创建历史快照。
- 汇总最终结果的识别信息。
- 更新结果状态和错误原因。
- 保存 run 的整体状态。
- 将 apply 记录裁剪到最近 100 条。
- 保持快照与 `operations` 同步写入。

不负责 GUI 展示结构。

### 5.2 `launcher_core.py`

提供纯数据转换能力：

- 读取并校验运行日志的历史视图结构。
- 过滤 `apply` 记录。
- 将 run 状态和结果状态转换为中文展示状态。
- 识别旧版无快照记录。
- 将日志错误转换为 GUI 可展示的数据结果。

该模块不修改日志、不检查当前目标路径，也不执行撤销。

### 5.3 `launcher_gui.py`

负责：

- 新增“执行历史”导航入口。
- 每次进入页面时调用读取逻辑。
- 展示历史列表、详情表格和展开内容。
- 展示空状态、旧记录状态和损坏日志错误。

禁止：

- 根据路径或名称推断历史结果。
- 修改或删除历史记录。
- 在历史页面发起撤销。
- 复制分类、合并、数量计算或命名业务逻辑。

## 6. 错误处理

历史快照是 apply 审计能力的一部分，但不能以破坏撤销日志为代价。快照和 `operations` 不使用彼此独立的文件写入流程。

- 首次创建 run 或首次持久化完整快照失败时，apply 不开始真实文件操作，并记录明确错误。
- 真实文件操作完成后，如果包含最新 `operations` 和快照状态的联合保存失败，保留替换前的完整日志，人工日志记录错误，命令返回失败。
- 发生上述执行中保存失败后，不继续执行后续结果，避免产生更多无法可靠记录撤销信息的文件操作。
- 只要历史快照或最终 run 状态未能可靠保存，run 不得被报告为 `success`。
- 任何日志保存失败都不得用空日志覆盖原日志。
- GUI 读取失败时只报告问题，不修改文件。

对于程序被强制终止的情况：

- run 可能保留 `running`。
- 快照结果可能保留 `pending`。
- GUI 分别显示为“执行中断”。
- 不尝试根据磁盘现状自动补写成功或失败。

## 7. 测试与验收

### 7.1 核心日志

- 新版 apply 写入 `history_snapshot.schema_version`。
- 单文件夹结果保存正确的名称、路径、来源和识别信息。
- 合并结果正确汇总日期范围、品类、单量、数量和关键词。
- 成功、跳过、失败结果状态及原因正确。
- 中断记录保留 `running` 和 `pending`，读取时显示为“执行中断”。
- `operations` 结构和现有撤销行为不变。

### 7.2 保留上限

- 第 100 条 apply 写入后不裁剪。
- 第 101 条 apply 写入后删除最旧记录。
- 旧版无快照记录计入上限。
- 被删除记录的 `operations` 同步消失，无法再被 `undo-last` 选中。
- `rename_log.csv` 不受裁剪影响。
- 裁剪后日志仍为完整有效 JSON。

### 7.3 历史读取

- 只返回 `apply` 记录。
- `success`、`partial`、`failed` 和 `running` 映射正确。
- 没有快照的旧记录标记为详情不完整。
- 日志不存在、空日志及无 apply 记录返回空状态。
- 损坏 JSON 返回错误和绝对路径，不返回部分记录。
- 读取过程不创建或修改日志文件。

### 7.4 GUI

- 左侧导航显示“执行历史”。
- 每次进入页面重新读取日志。
- 进入页面后不默认选择记录。
- 选择新版记录后按最终结果文件夹显示详情。
- 展开结果后显示来源项和按需显示的错误原因。
- 选择旧记录后显示详情不完整。
- 空状态和损坏日志状态符合设计。

### 7.5 回归验证

- `python -m py_compile file_helper.py launcher_core.py launcher_gui.py`
- 运行现有完整单元测试。
- 在临时根目录执行成功、合并、跳过和失败 apply 场景。
- 验证最近一次未淘汰 run 的 `undo-last`。
- 验证超过 100 条后最旧 run 不再可撤销。
- 启动 GUI 源码并检查历史页基本布局和展开交互。
- 如果用户要求发布包同步，重建并验证 foldered PyInstaller 发布目录；仅完成源码时必须明确说明发布包未更新。

## 8. 不在第一版范围

- dry-run 和 undo-last 历史。
- 跨运行搜索。
- 统计图表和趋势分析。
- 手动删除或清空历史。
- 打开来源或目标路径。
- 从历史页面执行撤销。
- 任意旧 run 撤销。
- 从 CSV、整理报告或当前磁盘重建旧详情。
- GUI 中显示 100 条保留上限提示。
