# 配置管理与自动更新设计

## 1. 目标

为 Windows 文件整理助手增加两项能力：

1. 在 GUI 中管理品类、关键词、排序、启用状态和合并开关。
2. 启动时检查 GitHub Releases，新版本由用户确认后自动下载和安装。

软件升级必须保留用户配置，不覆盖日志、启动器设置或用户文件。

本设计分两阶段实施：

- 阶段一：配置分层、旧配置迁移、GUI 配置管理。
- 阶段二：版本检查、下载校验、独立更新、失败回滚。

阶段一可以独立发布和验收，阶段二不得绕过阶段一的数据隔离机制。

## 2. 发布目录

继续使用免安装、解压即用的文件夹版：

```text
Windows文件整理助手\
├─ Windows文件整理助手.exe
├─ file_helper.py
├─ config.default.yaml
├─ user_config.yaml
├─ VERSION.txt
├─ updater.exe
└─ _internal\
```

- `config.default.yaml` 是官方配置，随版本更新。
- `user_config.yaml` 是用户差异配置，由 GUI 管理，更新器不得覆盖。
- 用户配置继续放在软件文件夹中，不使用 `%APPDATA%`。
- 软件放在无写权限目录时，GUI 必须明确提示无法保存，不得假装保存成功。

## 3. 配置分层

### 3.1 生效顺序

程序加载顺序：

```text
config.default.yaml
        +
user_config.yaml
        =
运行时最终配置
```

官方配置仍提供完整、可独立运行的默认规则。用户配置只保存差异，不复制整份官方配置。

### 3.2 用户配置内容

`user_config.yaml` 可以记录：

- 用户新增的品类。
- 用户给官方品类补充的关键词。
- 官方品类或官方关键词的停用状态。
- 官方品类的 `merge_enabled` 覆盖值。
- 自定义品类及关键词的启用状态。
- 用户调整后的完整品类排序。

官方品类名称不可删除或修改。自定义品类可以删除。

### 3.3 合并原则

- 同名官方品类使用官方定义作为基础，再应用用户差异。
- 自定义品类追加到最终品类集合。
- 用户设置的品类排序覆盖官方 `category_priority`；新版本新增且尚未出现在用户排序中的官方品类追加到末尾，避免升级导致已有顺序变化。
- 停用品类不参与识别、合并、排序和关键词冲突检查。
- 停用关键词不参与识别和关键词冲突检查。
- 用户修改官方品类的合并开关时，以用户值为准。
- 新增自定义品类的 `merge_enabled` 默认值为 `true`。
- 最终运行配置仍必须经过现有配置校验后才能用于预览或执行。

## 4. 旧配置迁移

第一次使用分层配置的版本时，可能存在用户手工修改过的旧 `config.yaml`。

迁移流程：

1. 保留旧 `config.yaml`，不覆盖、不删除。
2. 将旧配置与该旧版本对应的官方基准配置比较。
3. 只提取用户新增品类、用户新增关键词、合并开关变化和排序变化。
4. 将提取结果展示给用户确认。
5. 用户确认后写入 `user_config.yaml`。
6. 迁移成功后将原文件重命名为带时间戳的备份文件。

如果无法获得可靠的旧版官方基准，程序不得猜测哪些项目是用户修改。此时应提示用户进入配置页手工导入或整理差异。

## 5. GUI 配置页

### 5.1 导航与布局

在现有 GUI 左侧导航增加“配置管理”。右侧工作区分为：

1. 品类列表。
2. 当前品类详情。
3. 关键词批量添加区。
4. 固定的保存和状态区域。

`launcher_gui.py` 只负责界面交互。配置读取、合并、验证和序列化放入独立配置模块，避免把业务规则复制进 GUI。

### 5.2 品类列表

- 官方品类和自定义品类统一显示。
- 官方品类显示“官方”标识。
- 自定义品类显示“自定义”标识。
- 支持“上移”“下移”。
- 支持鼠标拖动排序。
- 品类具有滑动启用开关。
- 停用品类保留在原位置，整行变灰。
- 官方品类不可删除，只能停用。
- 自定义品类可以直接删除，不弹二次确认。
- 删除只改变当前编辑状态，点击保存后才写入文件。

### 5.3 品类详情

- 显示品类名称。
- 自定义品类名称可编辑；官方品类名称只读。
- 提供“允许同品类合并”滑动开关。
- 官方和自定义品类均允许修改合并开关。
- 新增自定义品类默认开启合并。

### 5.4 关键词管理

- 关键词以表格形式显示，一行一个。
- 每个关键词具有滑动启用开关。
- 停用关键词保留原位置并变灰。
- 官方关键词不可删除，只能停用。
- 用户补充的关键词和自定义品类关键词可以删除。
- 官方品类允许补充用户关键词。
- 停用官方关键词后，该关键词可以分配给其他启用品类。

### 5.5 批量输入

同时支持：

- 表格逐行新增。
- 批量文本输入。

批量文本支持英文逗号、中文逗号和换行分隔。导入时自动：

1. 去除首尾空格。
2. 丢弃空值。
3. 对同一批次去重。
4. 将结果加入关键词表格。

导入动作不直接写文件，仍需点击保存。

### 5.6 滑动开关

品类启用、关键词启用和允许合并均使用统一滑动开关：

- 绿色表示开启。
- 灰色表示关闭。
- 状态改变立即反映在界面预览中。
- 只有点击“保存配置”后才正式生效。

### 5.7 未保存修改

用户离开配置页或关闭软件时，如果存在未保存修改，显示：

- 保存
- 放弃
- 取消

“取消”保持当前页面和编辑状态。

## 6. 校验与冲突

保存前执行完整校验：

- 品类名称不能为空。
- 启用品类名称不能重复。
- 启用关键词不能为空。
- 同一启用品类内不得存在重复启用关键词。
- 同一启用关键词不得同时属于两个启用品类。
- 排序列表必须覆盖所有当前品类且不得重复。
- 合并后的最终配置必须通过现有模板和配置校验。

关键词冲突时：

- 禁止保存。
- 标出冲突关键词。
- 标出该关键词涉及的所有品类。
- 自动定位或允许跳转到相关品类。

停用品类和停用关键词不参与关键词冲突检查。重新启用导致冲突时，同样禁止保存。

保存必须采用“写入临时文件，再原子替换”的方式。写入失败时保留原 `user_config.yaml`。

## 7. 自动更新

### 7.1 检查时机

- GUI 启动完成后异步检查更新。
- 网络失败、超时或 GitHub 不可访问时静默保留正常功能，可在状态区域显示“检查更新失败”。
- 检查更新不得阻塞文件整理功能。
- 整理任务运行期间不得开始安装更新。

### 7.2 更新提示

发现更高版本后显示：

- 当前版本。
- 最新版本。
- 更新说明。
- “立即更新”和“稍后提醒”按钮。

不得静默安装。只有用户点击“立即更新”后才下载。

### 7.3 下载与校验

- 更新包从 GitHub Releases 下载到临时目录。
- 发布信息包含版本号、ZIP 地址和 SHA-256。
- 下载完成后必须核对 SHA-256。
- 校验失败时删除临时更新包并提示失败。
- 版本比较使用语义化版本规则，不使用普通字符串比较。

### 7.4 独立更新器

主程序不能直接覆盖正在运行的 EXE，因此由独立更新器执行：

1. 主程序确认当前没有整理任务。
2. 主程序启动更新器并传入进程号、更新包、安装目录和重启目标。
3. 主程序退出。
4. 更新器等待主程序完全退出。
5. 更新器备份将要替换的程序文件。
6. 更新器解压并替换允许更新的文件。
7. 更新器明确保留用户文件。
8. 更新成功后重启主程序。
9. 任一步骤失败时恢复备份，并显示错误。

### 7.5 永久保留项

更新器不得覆盖或删除：

- `user_config.yaml`
- `launcher_settings.json`
- `rename_log.csv`
- `organizer_run_log.json`
- 用户生成的整理报告
- 未列入更新清单的用户文件

更新包必须包含明确的文件清单。更新器不得采用“清空安装目录后整体复制”的策略。

## 8. 安全边界

- 自动更新不得改变现有 dry-run 默认行为。
- 自动更新不得追加或绕过 `--yes`。
- 配置页不得执行整理、移动、合并、压缩或撤销。
- 配置保存后只影响后续预览或执行。
- 更新失败不得阻止旧版本继续运行。
- 用户配置损坏时，程序应报告错误并允许打开配置页修复，不得静默丢弃。
- 官方配置和用户配置合并后仍遵守现有不覆盖目标、不删除原文件和确认优先规则。

## 9. 测试与验收

### 阶段一

- 官方配置单独加载与现有行为一致。
- 用户新增品类和关键词可以识别。
- 官方品类补充关键词可以识别。
- 停用品类和关键词不参与识别。
- 排序、合并开关覆盖正确。
- 关键词冲突禁止保存并准确标出品类。
- 上移、下移和拖动得到相同排序结果。
- 未保存离开提示正确。
- 保存失败不破坏旧用户配置。
- 旧配置迁移只提取可确认的用户差异。
- CLI、dry-run、apply、undo 的安全行为保持不变。

### 阶段二

- 无网络时正常启动和整理。
- 同版本不提示更新。
- 新版本提示正确。
- 用户拒绝后不下载、不安装。
- SHA-256 不匹配时拒绝安装。
- 更新期间用户配置、设置和日志保持不变。
- 模拟替换失败后可以恢复旧版本。
- 更新成功后重新启动并显示新版本。
- 整理任务运行期间不能安装更新。

## 10. 发布策略

阶段一发布后，先验证多个用户配置样本和旧配置迁移，再实施阶段二。自动更新首次上线时应保留现有手工下载发布方式，直到更新与回滚流程经过稳定验证。
